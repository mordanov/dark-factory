## Scope

service, in priority order for backend implementation:

1. `src/core/auth_adapter.py` (T012) — incoming JWT validation boundary
2. `src/core/security.py` (T011) — service-to-service JWT generation
3. `SERVICE_JWT` and TM credentials exposure (T032, T035, FR-013)
4. Secrets-in-logs guardrails (T013, FR-013)
5. Subprocess runner security (T030)
6. Context token injection safety (T032)

## 1. `src/core/auth_adapter.py` — Required Design (T012)

### Reference Implementation

Mirror `services/orchestrator/src/core/auth_adapter.py` exactly. The orchestrator's
pattern is:

```python
class AuthAdapter:
    def __init__(self, settings: Settings) -> None:
        if settings.auth_mode not in ("local", "keycloak"):
            raise ValueError(f"Unknown AUTH_MODE: {settings.auth_mode!r}")
        self.settings = settings

    async def verify(self, token: str) -> dict:
        if self.settings.auth_mode == "keycloak":
            raise NotImplementedError("Keycloak validation not implemented")
        return verify_access_token(token)
```

### Mandatory Requirements

**R-AUTH-01 (Blocker)**: `AUTH_MODE` validation MUST happen at `__init__` time, not at
first request. The service must fail at startup if `AUTH_MODE` is any value other than
`"local"` or `"keycloak"`. Never silently fall through to a permissive default.

Implementation check:
```python
# CORRECT — raises at startup if AUTH_MODE is unrecognised
def __init__(self, settings: Settings) -> None:
    if settings.auth_mode not in ("local", "keycloak"):
        raise ValueError(f"Unknown AUTH_MODE: {settings.auth_mode!r}")

# WRONG — falls through silently
def __init__(self, settings: Settings) -> None:
    self.settings = settings  # No validation
```

**R-AUTH-02 (Blocker)**: `AUTH_MODE=keycloak` MUST raise `NotImplementedError` on any
call to `verify()`. It MUST NOT silently accept tokens.

**R-AUTH-03 (High)**: `AUTH_MODE=local` MUST call `verify_access_token()` without
modification. Do not add any additional token relaxation (e.g., skipping expiry checks,
accepting tokens without a `"type"` claim).

**R-AUTH-04 (Medium)**: The `AuthAdapter` MUST be a FastAPI dependency injected via
`Depends()`. Do not instantiate it per-request without dependency injection — settings
reuse is intentional for startup-time validation.

Recommended FastAPI wiring:
```python
# src/api/v1/runs.py
from fastapi import Depends, HTTPException, Security
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

bearer = HTTPBearer()

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Security(bearer),
    auth: AuthAdapter = Depends(get_auth_adapter),
) -> dict:
    try:
        return await auth.verify(credentials.credentials)
    except (JWTError, NotImplementedError) as exc:
        raise HTTPException(status_code=401, detail="Invalid or missing token") from exc
```

**R-AUTH-05 (High)**: On `JWTError`, return HTTP 401. On `NotImplementedError`
(keycloak mode), return HTTP 501 — do NOT return 401 or 403, as that would imply the
request was authenticated but denied. 501 signals the configuration gap explicitly.

## 2. `src/core/security.py` — Service JWT Generation (T011)

### Reference Implementation

Mirror `services/orchestrator/src/core/security.py`. The new service only adds:
- `create_service_token()` — for outbound calls (to Orchestrator, TM, ContextDistiller)
- `verify_access_token()` — for inbound call validation

### Mandatory Requirements

**R-SEC-01 (Blocker)**: `create_service_token()` MUST use `sub="service:agent-dispatcher"`
to distinguish service tokens from user tokens. The subject must be namespaced with
`"service:"` prefix — consistent with monorepo convention.

**R-SEC-02 (High)**: Service tokens MUST have a short expiry. Recommended:
`SERVICE_JWT_EXPIRE_HOURS` setting, default 1 hour. Do not use `ACCESS_TOKEN_EXPIRE_MINUTES`
(designed for user sessions, not service calls).

**R-SEC-03 (High)**: `verify_access_token()` MUST check the `"type"` claim equals
`"access"` — mirroring orchestrator's pattern. Service tokens generated by
`create_service_token()` do NOT need a `"type"` claim (they have a `"service:"` sub),
and incoming user tokens are validated for type. Keep these two paths distinct.

```python
def create_service_token() -> str:
    now = datetime.utcnow()
    expire = now + timedelta(hours=settings.service_jwt_expire_hours)
    payload = {
        "sub": "service:agent-dispatcher",
        "exp": expire,
        "iat": now,
    }
    return jwt.encode(payload, settings.jwt_secret_key, algorithm=settings.jwt_algorithm)

def verify_access_token(token: str) -> dict:
    try:
        payload = jwt.decode(token, settings.jwt_secret_key, algorithms=[settings.jwt_algorithm])
    except JWTError as exc:
        raise exc
    if payload.get("type") != "access":
        raise JWTError("Not an access token")
    return payload
```

**R-SEC-04 (Medium)**: Do NOT cache `create_service_token()` output as a module-level
global. Generate a fresh token per outbound HTTP call to avoid sending expired tokens
after the dispatcher has been running for hours.

## 3. SERVICE_JWT and Credential Exposure (T032, T035, FR-013)

This is the highest-risk area unique to this service. The context document injected into
every agent includes a `SERVICE_JWT` that grants the agent access to Ticket Manager and
Orchestrator APIs. This token MUST NEVER be persisted.

### Trust Boundary

```
┌─────────────────────────────────────────────────────┐
│  EPHEMERAL — only in agent process memory and stdout │
│  ┌───────────────────────────────────────────────┐  │
│  │ Context text (agent input)                    │  │
│  │   includes: SERVICE_JWT, TM_BASE_URL, creds   │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
         │ Flows through subprocess stdout / API response
         ▼
┌──────────────────────────────────────────────┐
│  STRIPPED BEFORE PERSISTENCE                 │
│  agent_runs.context_snapshot  ← NO JWT       │
│  agent_runs.raw_output        ← NO JWT       │
│  API responses                ← NO JWT       │
└──────────────────────────────────────────────┘
```

### Mandatory Requirements

**R-JWT-01 (Blocker)**: `context_builder.py` MUST separate the context text (containing
`SERVICE_JWT`) from the `AgentContext` schema used for `context_snapshot` persistence.
The `AgentContext` Pydantic model MUST NOT have a `service_jwt` field.

The split:
```python
# CORRECT
async def build_context(ticket, agent_id, ...) -> tuple[str, AgentContext]:
    service_jwt = create_service_token()
    context_text = assemble_markdown(ticket, agent_id, service_jwt, ...)  # has JWT
    context_snapshot = AgentContext(ticket_id=ticket.id, ...)              # no JWT
    return context_text, context_snapshot

# WRONG
async def build_context(ticket, agent_id, ...) -> str:
    service_jwt = create_service_token()
    context = AgentContext(ticket_id=ticket.id, service_jwt=service_jwt, ...)  # JWT persisted!
    return context.model_dump_json()
```

**R-JWT-02 (Blocker)**: `dispatcher_service.py` MUST call `AgentRunRepository.create()`
with the `context_snapshot` dict (no JWT). The `context_text` string (containing JWT) is
passed only to `runner.run()` and discarded after the run completes.

```python
# CORRECT in dispatcher_service.py
context_text, context_snapshot = await context_builder.build_context(ticket, agent_id)
run = await run_repo.create(
    ticket_id=ticket.id,
    agent_id=agent_id,
    context_snapshot=context_snapshot.model_dump(),  # no JWT here
    ...
)
exit_code, stdout = await runner.run(agent_id, system_prompt, context_text, timeout)
# context_text is now ephemeral — not stored anywhere
```

**R-JWT-03 (Blocker)**: `raw_output` from agent stdout MUST be scanned for the JWT
before storage. The agent's stdout may echo back the context it received. Strip any
occurrence of the JWT from `raw_output` before calling `mark_done()` or equivalent.

Implementation:
```python
def strip_service_jwt(text: str, jwt_value: str) -> str:
    """Remove the service JWT from captured agent output before storage."""
    if not jwt_value or jwt_value == "null":
        return text
    return text.replace(jwt_value, "[SERVICE_JWT_REDACTED]")

# In dispatcher_service.py, after runner.run():
safe_output = strip_service_jwt(stdout, service_jwt)
await run_repo.mark_done(run.id, safe_output, parsed_result)
```

**R-JWT-04 (High)**: API responses from `GET /api/v1/runs` and `GET /api/v1/runs/{id}`
MUST NOT include the JWT. Since the JWT is excluded from `context_snapshot` (R-JWT-01),
this is enforced at the schema level — but verify the `raw_output` field also cannot
contain the JWT at response time (covered by R-JWT-03).

**R-JWT-05 (High)**: TM service credentials (username/password) used to POST comments
MUST NOT appear in `context_snapshot`, `raw_output`, `error_message`, or log output.
The `reporter.py` module uses a service JWT for TM calls — not the credentials file.
Ensure no credential values are interpolated into error messages.

## 4. Secrets-in-Logs Guardrails (T013, FR-013)

**R-LOG-01 (Blocker)**: The following values MUST NEVER appear in application logs:
- `SERVICE_JWT` value (any JWT string generated by `create_service_token()`)
- `TM_PASSWORD` / `TM_SERVICE_CREDENTIAL` values
- `JWT_SECRET_KEY` value
- `POSTGRES_PASSWORD` / `DATABASE_URL` with embedded password

**R-LOG-02 (High)**: When logging agent run context in `dispatcher_service.py`, log
only the `ticket_id`, `agent_id`, and `runner_mode` — never the full `context_text`.
Use structured log fields:
```python
logger.info(
    "Starting agent run",
    extra={"ticket_id": ticket.id, "agent_id": agent_id, "runner_mode": runner_mode}
)
# NOT:
logger.info(f"Context for agent: {context_text}")  # would expose JWT
```

**R-LOG-03 (High)**: Exception tracebacks MUST NOT include JWT or credential values.
Catch and re-raise with sanitized messages:
```python
except Exception as exc:
    # Never: logger.error(f"Context build failed: {exc}", exc_info=True)
    # Safe:
    logger.error("Context build failed for ticket %s: %s", ticket_id, type(exc).__name__)
    raise
```

**R-LOG-04 (Medium)**: `DATABASE_URL` MUST be constructed without embedding credentials
in logged startup messages. If logging is added to `session.py`, log only the host/port,
not the full URL:
```python
# WRONG
logger.info(f"Connecting to {settings.database_url}")  # exposes password

# CORRECT
from urllib.parse import urlparse
parsed = urlparse(settings.database_url)
logger.info("Connecting to %s:%s/%s", parsed.hostname, parsed.port, parsed.path.lstrip('/'))
```

**R-LOG-05 (Medium)**: The settings object MUST override `__repr__` or use Pydantic's
secret field type for sensitive values, so accidental `str(settings)` calls in logs
do not expose secrets. Recommended: use `SecretStr` for `JWT_SECRET_KEY` and
`POSTGRES_PASSWORD` in `config.py`.

## 5. Subprocess Runner Security (T030)

**R-PROC-01 (Blocker)**: `ClaudeCodeRunner` MUST use `asyncio.create_subprocess_exec`,
NOT `create_subprocess_shell`. Shell=True would allow command injection if `agent_id`
were ever derived from untrusted input.

```python
# CORRECT — no shell interpolation
proc = await asyncio.create_subprocess_exec(
    settings.claude_code_path,
    "--print",
    "--mcp-config", str(settings.claude_mcp_config_path),
    "--system-prompt", system_prompt,
    context,           # positional argument to claude
    stdout=asyncio.subprocess.PIPE,
    stderr=asyncio.subprocess.PIPE,
)

# WRONG — shell=True enables injection
proc = await asyncio.create_subprocess_shell(
    f"{settings.claude_code_path} --print {context}",  # injection risk
    stdout=asyncio.subprocess.PIPE,
)
```

**R-PROC-02 (High)**: Agent `system_prompt` content is loaded from disk via
`AGENT_PROMPTS_DIR/{agent_id}.md`. The `agent_id` value comes from the Orchestrator
response. Validate `agent_id` against a whitelist before constructing the path to
prevent path traversal:

```python
ALLOWED_AGENT_IDS = frozenset({
    "backend", "frontend", "software_architect", "security_architect",
    "project_manager", "designer", "code_reviewer", "autotester",
    "devops", "project_administrator"
})

def get_prompt_path(agent_prompts_dir: Path, agent_id: str) -> Path:
    if agent_id not in ALLOWED_AGENT_IDS:
        raise PromptNotFoundError(f"Unknown agent_id: {agent_id!r}")
    path = agent_prompts_dir / f"{agent_id}.md"
    # Resolve and check the path stays within prompts dir
    resolved = path.resolve()
    if not str(resolved).startswith(str(agent_prompts_dir.resolve())):
        raise PromptNotFoundError(f"Path traversal attempt: {agent_id!r}")
    return path
```

**R-PROC-03 (Medium)**: Subprocess timeout handling MUST kill the process group, not
just the subprocess PID, to avoid zombie processes. Use `proc.kill()` followed by
`await proc.communicate()` to drain buffers:
```python
try:
    stdout, stderr = await asyncio.wait_for(proc.communicate(), timeout_seconds)
except asyncio.TimeoutError:
    proc.kill()
    stdout, stderr = await proc.communicate()  # drain to avoid pipe deadlock
    return (-1, (stdout or b"").decode("utf-8", errors="replace")[:64_000])
```

## 6. Context Token Injection Safety (T032)

**R-CTX-01 (Medium)**: Project memory from ContextDistiller is untrusted external data.
When injecting it into the agent context, do NOT interpret it as instructions or allow
it to override the agent's system prompt. Inject it within a clearly delimited section:
```markdown
## Project Context
<!-- BEGIN UNTRUSTED PROJECT MEMORY -->
{project_memory_content}
<!-- END UNTRUSTED PROJECT MEMORY -->
```

**R-CTX-02 (Medium)**: `CONTEXT_MAX_TOKENS` truncation MUST apply to `project_memory`
before injection. Verify the truncation implementation truncates at a word/sentence
boundary and cannot be bypassed by a maliciously long memory document.

**R-CTX-03 (Low)**: ADR content from ContextDistiller is also untrusted. Apply the
same section delimiting pattern as project memory.

## 7. NOTIFY Trigger Security Note

**R-DB-01 (Informational)**: The `pg_notify('df_new_agent_run', row_to_json(NEW)::text)`
trigger fires on every `INSERT` to `agent_runs`. If any monitoring consumer listens
on this channel, ensure the notification payload does NOT expose `context_snapshot`
fields that could be used by an attacker with DB notification access. Since
`context_snapshot` excludes the JWT (R-JWT-01), the risk is limited to ticket metadata.
Consider stripping `context_snapshot` from the NOTIFY payload if a listener is added.

## Review Decision

**Status**: APPROVED WITH CONDITIONS

**Conditions (must be met before Phase 3 merges)**:

1. `auth_adapter.py` raises `ValueError` at init for unknown `AUTH_MODE` — not at request time.
2. `auth_adapter.py` raises `NotImplementedError` for keycloak, returns HTTP 501.
3. `context_builder.py` returns `(context_text, context_snapshot)` — JWT in text only, never in snapshot.
4. `dispatcher_service.py` calls `strip_service_jwt(stdout, jwt)` before persisting `raw_output`.
5. `agent_id` whitelist validation before path construction in prompt loader.
6. `create_subprocess_exec` (not shell) in `ClaudeCodeRunner`.
7. Security test cases SEC-01 through SEC-08 written and passing.

**Non-blocking (track post-Phase 3)**:
- `SecretStr` for sensitive config fields (R-LOG-05) — Low risk, developer ergonomics.
- NOTIFY trigger payload stripping (R-DB-01) — Only relevant if a monitoring consumer is added.
- Structured logging (R-LOG-02, R-LOG-03) — Medium, address before production.
