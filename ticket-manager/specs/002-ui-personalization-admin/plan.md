# Implementation Plan: UI Personalization & Admin Management

**Branch**: `002-ui-personalization-admin` | **Date**: 2026-05-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/002-ui-personalization-admin/spec.md`

## Summary

Four improvements to the ticket management application: (1) three-language i18n using `react-i18next`; (2) admin user management page with create/edit/block/unblock; (3) URL-persisted board/list view state via `?view=` query param; (4) six WCAG 2.1 AA-compliant color themes via CSS custom properties. Backend changes are limited to a `blocked_at` column on `users`, new admin API endpoints, and a login-time block check. All other changes are frontend-only.

## Technical Context

**Language/Version**: Python 3.11 (backend), TypeScript 5 / React 18 (frontend)
**Primary Dependencies**: FastAPI 0.115.x, SQLAlchemy 2.0.x async, Alembic, Pydantic v2 (backend); React Router DOM v6, React Query v5, Zustand, i18next v24, react-i18next v15 (frontend вЂ” i18next to be installed)
**Storage**: PostgreSQL 15 (backend data); `localStorage` (frontend language + theme preferences)
**Testing**: pytest + pytest-asyncio (backend); Vitest + Testing Library (frontend)
**Target Platform**: Web (desktop-primary, mobile-responsive)
**Project Type**: Web service (FastAPI backend + React SPA frontend)
**Performance Goals**: Language switch < 500ms; theme switch < 300ms (both are synchronous CSS/JS ops вЂ” trivially achievable)
**Constraints**: No email service integration; admin sets passwords directly. No WebSocket or polling for session invalidation (next-login enforcement only).
**Scale/Scope**: Single-tenant internal tool; user count in tens to hundreds.

## Implementation Notes

### US1 вЂ” Multi-Language Interface

- Install `i18next` and `react-i18next` as runtime dependencies.
- Create `src/i18n.ts` with `i18next.init()` вЂ” reads `localStorage.getItem("lang") ?? "en"`, loads the matching locale bundle.
- Wrap `<App />` in `<I18nextProvider i18n={i18n}>` in `main.tsx`.
- Replace all hardcoded JSX strings with `t("key")` calls. Static label maps (`TICKET_STATUS_LABELS`, etc.) become `t()` lookups.
- `LanguageSwitcher` writes `localStorage.setItem("lang", code)` and calls `i18n.changeLanguage(code)`.
- Translation keys live in `src/locales/{lang}/common.json`. All three files must be complete before merge.

### US2 вЂ” Admin User Management

- **Backend**: New `src/api/v1/admin.py` router mounted at `/admin`; all routes require `require_role("administrator")`. New `admin_service.py` contains business logic and `structlog` audit calls.
- **Login block check**: In `src/services/auth_service.py` (or the login endpoint), after validating password, check `user.blocked_at is not None` в†’ raise HTTP 403 with `"Your account has been blocked. Contact an administrator."`.
- **Self-protection**: Service layer checks `actor.id == target.id` and raises HTTP 403 for block/edit-own-role actions.
- **Frontend**: `AdminUsersPage` uses React Query to fetch `/api/v1/admin/users`. Mutations for create, edit, block, unblock invalidate the `["admin", "users"]` query key.
- Route `/admin/users` added to `router.tsx`, wrapped in a role-gated `ProtectedRoute` variant that checks `currentUser.role === "administrator"`.

### US3 вЂ” Persistent URL Navigation

- In `ProjectPage.tsx`: replace `const [view, setView] = useState<View>("list")` with `useSearchParams()`. Read `searchParams.get("view") as View ?? "list"`. Write via `setSearchParams({ view: newView }, { replace: true })`.
- No backend changes. No router changes (path stays `/projects/:projectId`).

### US4 вЂ” Color Scheme Personalization

- `src/styles/themes.css` defines CSS custom properties under selectors `[data-theme="light"]` (default on `:root`), `[data-theme="dark"]`, etc.
- `useTheme` hook: on mount, reads `localStorage.getItem("theme") ?? "light"`, sets `document.documentElement.dataset.theme = theme`. On change, updates both localStorage and the DOM attribute.
- `ThemeSwitcher` component renders 6 labelled swatches. Clicking one calls `setTheme(key)` from `useTheme`.
- All existing components use CSS variables from `variables.css`; no per-component changes needed beyond importing the variables file.
