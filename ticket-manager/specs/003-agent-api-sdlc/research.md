# Research: Agent API SDLC Integration

**Feature**: `003-agent-api-sdlc` | **Date**: 2026-05-24

## Decision 1: Atomic resource increment strategy

**Decision**: Use `UPDATE tickets SET time_spent = time_spent + :delta WHERE id = :id` via SQLAlchemy `update()` with `Ticket.time_spent + delta` expression, executed inside the existing async session (no extra transaction needed). Combined with `WITH FOR UPDATE` row lock (consistent with the pattern in `transition_service.py`).

**Rationale**: PostgreSQL handles the add-in-place atomically. Using a `+= delta` expression avoids read-modify-write races without needing application-level distributed locking. The row lock ensures that concurrent increments serialize correctly.

**Alternatives considered**:
- Application-level read-modify-write: rejected — races under concurrent agent submissions.
- Separate `resource_increments` table (append-only, aggregate on read): over-engineered for current scale; the `TicketEvent` journal already provides the immutable audit trail.

## Decision 2: Mandatory update text mechanism

**Decision**: Reuse the existing `ProgressUpdate` mechanism. Before an agent transitions a ticket, it calls `PUT /api/v1/tickets/{id}/progress` with the update text. The transition service already enforces that all assignees must have submitted at least one progress update (the "progress gate"). No changes to the transition endpoint's input schema are needed.

**Rationale**: The existing gate already satisfies FR-011 ("transition MUST include a non-empty text update") because it blocks the transition until all assignees have submitted progress. Duplicating the text in the transition request body would be redundant.

**Alternatives considered**:
- Inline `update_text` field on `TransitionRequest`: creates two codepaths for the same data; rejected in favor of the existing gate.

## Decision 3: Transition RBAC enforcement

**Decision**: Add a 4-line check in `transition_service.transition_ticket` after loading `all_assignments`. If `actor.id` is not in `{a.user_id for a in all_assignments}`, raise `HTTP 403 Forbidden` with detail `"Only assignees may transition this ticket"`. This applies to all roles including `administrator`.

**Rationale**: The spec explicitly forbids the project administrator from transitioning tickets unless assigned. Using the same code path for all roles keeps the enforcement simple and consistent.

**Alternatives considered**:
- Role-specific bypass for administrator: explicitly rejected by spec.
- Middleware-based check: unnecessary complexity for a single-endpoint rule.

## Decision 4: Admin password reset

**Decision**: Add `password: str | None = Field(None, min_length=8)` to `AdminUserUpdate`. In `admin_service.update_user`, if `password` is provided, hash it with `hash_password()` and update `user.hashed_password`. Emit a structured log event `admin_user_password_reset`.

**Rationale**: The bootstrapping flow needs to reset a known user's password when stored credentials are found to be wrong. Extending the existing PATCH endpoint is minimal and auditable.

**Alternatives considered**:
- Separate `POST /admin/users/{id}/reset-password` endpoint: adds a new route for one field; the PATCH pattern is already established.
- Delete + recreate user: loses user history and event references; rejected.

## Decision 5: Credential file format and location

**Decision**: Each agent's credential file is `{role-dir}/credentials.json` with schema `{"username": "<email>", "password": "<plaintext>"}`. Email convention: `{role}@agents.local`. Password: 24-character random alphanumeric string generated by the project-administrator skill using Python `secrets.token_urlsafe(18)`.

**Rationale**: Flat JSON is the simplest format readable by Claude Code with the `Read` tool. The `@agents.local` domain is clearly non-routable and distinguishes agent accounts from human accounts. `secrets.token_urlsafe` produces cryptographically random passwords.

**Alternatives considered**:
- YAML format: no advantage over JSON for this flat structure.
- Environment variables: not portable across terminal instances.
- UUID passwords: less readable for debugging; rejected in favor of URL-safe base64.

## Decision 6: `.gitignore` pattern

**Decision**: Add the following to `.gitignore`:
```
# Agent credential files (written by project-administrator skill)
*/credentials.json
```
This covers all `{role}/credentials.json` files at one directory depth. The project-administrator's own `credentials.json` is also covered.

**Rationale**: A glob pattern is simpler to maintain than enumerating every role folder explicitly. One-depth glob (`*/`) is safe — no deep source file will accidentally be named `credentials.json`.

**Alternatives considered**:
- Per-folder entries: verbose, breaks when a new agent is added.
- Top-level `credentials.json`: doesn't cover subdirectories.

## Decision 7: Agent skill file update strategy

**Decision**: Update only `agents/project-administrator.md` with the full bootstrapping workflow. For other agent skill files, add a short "Platform Authentication" section describing how to read credentials and authenticate. The `run-agents.sh` script is not modified — agents read their own credential files independently.

**Rationale**: The bootstrap workflow is complex enough to document in the skill file. Other agents need only a short authentication section. `run-agents.sh` is not changed because it already creates the per-role working directories (`mkdir -p`) and launches each agent in its working directory.

**Alternatives considered**:
- Inject credentials via environment variables in `run-agents.sh`: agents cannot rely on environment variables surviving across subprocess launches in Terminal.app.
- Central credential store: unnecessary indirection; each agent reads its own file.
