# Implementation Plan: Agent API SDLC Integration

**Branch**: `003-agent-api-sdlc` | **Date**: 2026-05-24 | **Spec**: [spec.md](spec.md)

## Summary

Add two resource-tracking fields (`time_spent`, `tokens_consumed`) to tickets and expose them via an increment-only API endpoint. Enforce assignee-only transition authorization in the existing transition service. Extend the admin API to support password reset. Update the `project-administrator` agent skill to bootstrap agent user accounts on the ticket platform by reading/writing JSON credential files in each agent's working folder. All credential files are excluded from version control.

## Implementation Notes

### Existing behavior to be aware of

- **Transition progress gate**: `transition_service.py` already blocks transitions if any assignee has not submitted a `ProgressUpdate`. This implements the "mandatory update before transition" requirement. No change needed here вЂ” the progress update IS the required text update.
- **Follow-up ticket support**: `Ticket.parent_ticket_id` already exists and the API already supports creating tickets with a parent reference. No changes needed.
- **Actual status enum values**: `TicketStatus.OPEN`, `IN_PROGRESS`, `IN_REVIEW`, `DONE`, `CLOSED`. The clarified spec used human-readable names (Open/In Progress/Review/Done) which map to these enum values. `CLOSED` exists as a terminal state beyond `DONE`.
- **WORKFLOW_TRANSITIONS**: allows `IN_REVIEW в†’ IN_PROGRESS` and `DONE в†’ IN_PROGRESS` (retry paths). This is pre-existing behavior вЂ” do not change it.
- **Existing admin API**: `POST /api/v1/admin/users` creates users; `PATCH /api/v1/admin/users/{id}` updates role/email but not password. Password reset requires extending `AdminUserUpdate`.

### Agent credential bootstrapping flow

```
project-administrator starts
  в†’ read project-administrator/credentials.json  (provided by human)
  в†’ POST /api/v1/auth/token  в†’ get admin JWT
  в†’ GET /api/v1/admin/users  в†’ get existing user list
  в†’ for each agent role in [product-manager, software-architect, ...]:
      email = "{role}@agents.local"
      if no credentials.json in role folder:
        generate_password()
        POST /api/v1/admin/users {email, password, role: "user"}
        write {username: email, password} в†’ {role}/credentials.json
      else:
        read {role}/credentials.json
        POST /api/v1/auth/token {email, password}
        if 401:
          generate_password()
          PATCH /api/v1/admin/users/{user_id} {password: new_password}
          update {role}/credentials.json
```
