# Quickstart: Frontend Design System

End-to-end walkthrough for verifying the design system is correctly installed and for adding new
UI following design system conventions.

---

## Prerequisites

- Node.js 18+
- `frontend/` dependencies installed: `npm install` from `frontend/`
- Backend running at `http://localhost:8000` (or mock mode)

---

## Scenario 1: Verify installation

After completing setup tasks (Tailwind + shadcn/ui install):

```bash
cd frontend
npm run dev
```

Open `http://localhost:5173`.

✅ Checks:
- Page renders without console errors
- No visible flash of unstyled content on load
- Font uses system sans-serif (not browser default serif)
- `<html>` has `data-theme="light"` attribute

---

## Scenario 2: Verify dark mode toggle

1. In the top navigation bar, click the sun/moon icon button.
2. Verify:
   - Page background transitions to dark immediately (no reload)
   - `<html>` now has both `data-theme="dark"` AND `class="dark"` (inspect in DevTools)
   - All surfaces (cards, inputs, modals, navbar) use dark palette — no white flash or partial light elements
3. Reload the page — dark mode persists.
4. Click the toggle again — reverts to light mode.

---

## Scenario 3: Verify keyboard navigation

1. Load the ticket list page.
2. Press `Tab` to move focus.
3. Verify:
   - Every button, link, and input receives a visible blue focus ring.
   - No element is skipped.
4. Open any modal (e.g., assign user).
5. While modal is open, press `Tab` — verify focus stays inside the modal.
6. Press `Escape` — modal closes and focus returns to the trigger button.

## Scenario 4: Verify responsive layout

1. In Chrome DevTools, toggle device simulation to iPhone 12 (390×844).
2. Navigate to the ticket list.
3. Verify:
   - No horizontal scrollbar
   - Navbar collapses or shows hamburger menu
   - Ticket cards are full-width and readable
4. Open a ticket detail page.
5. Verify:
   - Status transition buttons wrap properly
   - All form controls are full-width and tap-friendly

## Scenario 5: Adding a new shadcn/ui component

When you need a component not yet in `src/components/ui/`:

```bash
# From the frontend/ directory
npx shadcn@latest add tooltip
```

This generates `src/components/ui/tooltip.tsx`. Import and use immediately:

```typescript
import { Tooltip, TooltipContent, TooltipProvider, TooltipTrigger } from "@/components/ui/tooltip";

function MyFeature() {
  return (
    <TooltipProvider>
      <Tooltip>
        <TooltipTrigger asChild>
          <Button variant="ghost" size="icon">
            <InfoIcon className="h-4 w-4" />
          </Button>
        </TooltipTrigger>
        <TooltipContent>
          <p>Helpful info</p>
        </TooltipContent>
      </Tooltip>
    </TooltipProvider>
  );
}
```

✅ Rules:
- Do NOT edit files in `src/components/ui/` — these are regenerated by shadcn CLI.
- Wrap `ui/` primitives in domain-specific components if customization is needed.

## Scenario 6: Writing a new domain component

```typescript
// src/components/tickets/MyNewComponent.tsx
import { cn } from "@/lib/utils";
import { Card, CardContent, CardHeader, CardTitle } from "@/components/ui/card";
import { Badge } from "@/components/ui/badge";
import { Button } from "@/components/ui/button";

interface MyNewComponentProps {
  title: string;
  status: string;
  onAction: () => void;
  className?: string;
}

export function MyNewComponent({ title, status, onAction, className }: MyNewComponentProps) {
  return (
    <Card className={cn("hover:shadow-md transition-shadow", className)}>
      <CardHeader className="pb-2">
        <div className="flex items-center justify-between">
          <CardTitle className="text-base font-semibold">{title}</CardTitle>
          <Badge variant="secondary">{status}</Badge>
        </div>
      </CardHeader>
      <CardContent>
        <Button variant="outline" size="sm" onClick={onAction}>
          Take action
        </Button>
      </CardContent>
    </Card>
  );
}
```

✅ Rules:
- No `style` prop anywhere (FR-014).
- Use `cn()` for conditional/merged classes.
- Accept `className` prop for external overrides.
- Use `ui/` primitives for all visual elements.

## Scenario 7: Verify zero accessibility violations

```bash
# Install axe-core dev dependency if not present
npm install --save-dev axe-core @axe-core/react

# Run in development — axe logs violations to browser console
# Add to main.tsx (dev only):
if (process.env.NODE_ENV !== "production") {
  const axe = await import("@axe-core/react");
  axe.default(React, ReactDOM, 1000);
}
```

Or use the axe DevTools browser extension. Navigate every page and confirm zero violations.

## Scenario 8: Run existing tests after refactor

```bash
cd frontend
npm run test
```

All tests MUST pass (SC-007). If a test fails due to a missing `style` attribute or changed
class names, update the test assertion — but verify the rendered output is functionally equivalent
before updating.
